## 二、核心架构决策

### 决策 1：Reactive + Blocking JDBC 混合架构

| 层面 | 决策 | 原因 |
|------|------|------|
| Web | 保留 WebFlux，不加 spring-boot-starter-web | SSE流式输出是核心价值，切换回Servlet需重写Flux链路 |
| 数据访问 | Spring Data JPA（blocking） | Spring AI 的 JdbcChatMemoryRepository 基于 JdbcTemplate |
| 线程管理 | Blocking 操作在 `Schedulers.boundedElastic()` | 已验证模式（Phase 1 的 Graph 已在 boundedElastic 上执行） |
| 安全 | Reactive Spring Security（@EnableWebFluxSecurity） | WebFlux原生，SecurityWebFilterChain + WebFilter |

### 决策 2：安全上下文不依赖 Reactor Context 传播

`ReactiveSecurityContextHolder` 在 `Flux.create` + `subscribeOn(boundedElastic)` 链中不会自动传播。

**方案**：在 Controller 层提取 `JwtUser`，直接作为方法参数下传所有 Service/Graph。

### 决策 3：前端路由 + Token 管理

- 使用 `react-router-dom` 实现路由
- Token 存入 `localStorage`，API 请求通过 `Authorization: Bearer <token>` 携带
- SSE 流式请求使用 `fetch` + `ReadableStream`（非 `EventSource`），支持自定义 Header

## 六、关键风险与应对

| 风险 | 等级 | 应对 |
|------|------|------|
| Blocking JPA 在 reactive 线程池执行 | 🔴 高 | 所有 DB 调用强制 `Mono.fromCallable().subscribeOn(boundedElastic())` |
| JdbcChatMemory 表结构与 Spring AI 1.1.2 不匹配 | 🟡 中 | 先检查依赖源码，不匹配则用 JdbcTemplate 自定义实现 |
| SecurityContext 在 Flux.create 中丢失 | 🟡 中 | **不在 reactive 链中获取**，Controller 层提取后参数传递 |
| Token localStorage 存在 XSS 风险 | 🟢 低 | 阶段二可接受，生产化建议迁至 HTTP-only Cookie |
| SSE 流式需携带 Token | 🟢 低 | 已使用 fetch+ReadableStream，支持自定义 Header |

## 七、验收标准

1. ✅ 用户注册 → 登录 → 返回 JWT，Token 过期后请求被拒
2. ✅ 用户 A 的任务，用户 B 无法查看/修改（403）
3. ✅ Conversation 全生命周期 CRUD 正常，权限校验完整
4. ✅ ChatMemory 持久化正常，统一 conversation 的历史可被 Agent 引用
5. ✅ 恶意 Prompt 注入被 InputFilterService 拦截，返回明确错误
6. ✅ 前端登录 → 任务列表 → 创建任务 → SSE 流式 → 查看结果，全链路跑通
7. ✅ 多浏览器同时登录不同用户，数据完全隔离
